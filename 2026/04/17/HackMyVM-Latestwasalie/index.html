这绝对是一个非常经典的“踩坑”时刻！在 CTF 或者实际的安全研究中，因为 /etc/ld.so.preload 导致靶机或系统崩溃（甚至不得不重装）是极常发生的事情。你遇到的这个情况叫做无意中触发了“Fork 炸弹”（Fork Bomb）。

下面为你详细拆解为什么系统会崩掉，以及终端里那些报错是怎么来的：

1. 致命的连锁反应：无限套娃

你使用的提权手法核心思路是：将恶意动态链接库（pe.so）写入 /etc/ld.so.preload。 与环境变量 LD_PRELOAD 只作用于单次命令不同，/etc/ld.so.preload 是全局生效的。这意味着系统上每一次启动新进程，都会强制先加载你的 pe.so 并执行里面的 _init() 函数。

问题出在 pe.c 代码中的这一句：

C

在 C 语言中，system() 函数的底层实现是派生（fork）出一个新的子进程，并调用 /bin/sh -c "你的命令" 来执行。

于是，恐怖的死循环开始了：

  1. 你在终端敲了任何一个命令（比如 id 或 ls）。
  2. 系统加载 pe.so，执行 _init()。
  3. _init() 运行到 system()，系统派生出一个新的 /bin/sh 进程去执行 chmod。
  4. 因为启动了新的 /bin/sh 进程，系统再次读取 /etc/ld.so.preload，并把 pe.so 加载到这个新的 /bin/sh 进程中。
  5. 这个新的进程又触发了 _init()。
  6. _init() 再次调用 system()，又派生出一个新的 /bin/sh……

几百毫秒内，系统就会裂变出成千上万个 /bin/sh 进程。

2. 报错信息解读

终端里出现的报错完美印证了这个过程：

  • sh: 1: Cannot fork 这是系统发出的哀嚎。因为进程无限裂变，瞬间耗尽了系统允许的最大进程数量（PID 耗尽）或内存。系统无法再创建任何新进程，此时哪怕你敲 ls 或者试图杀掉进程都没用了，机器彻底假死卡崩。
  • chmod: cambiando los permisos de '/bin/bash': Operación no permitida (操作不允许) 这是因为 /etc/ld.so.preload 影响的是所有用户的所有进程。如果当前触发该库的进程不是 root 权限（比如你用普通用户 backupusr 执行的命令，或者系统后台的其他低权限守护进程），它也会去执行 chmod u+s /bin/bash，但由于没有 root 权限，就会报错“权限不足”。

3. 为什么代码里的 unsetenv 没拦住？

你的代码里有一句：unsetenv("LD_PRELOAD"); 这行代码的作用是清除 LD_PRELOAD 环境变量以防止死循环。但是，它防不住 /etc/ld.so.preload 文件！ 环境变量被清空了，但系统加载器在启动新进程时，依然会傻傻地去读取那个全局配置文件，导致防御机制失效。

LD_PRELOAD 与 LD_LIBRARY_PATH

LD_PRELOAD 环境变量用于指定一个或多个共享库（.so 文件），让加载器在加载所有其他库（包括标准 C 库 libc.so）之前优先加载它们。这个过程被称为“预加载”（preloading）库。

然而，为了维护系统安全并防止该功能被恶意利用（特别是在处理 suid/sgid 可执行文件时），系统会强制执行以下限制条件：

  • 当可执行文件的真实用户 ID（ruid）与有效用户 ID（euid）不匹配时，加载器会忽略 LD_PRELOAD。
  • 对于具有 suid/sgid 权限的可执行文件，只有位于标准路径下且同样具有 suid/sgid 权限的库才会被预加载。

如果你有权使用 sudo 执行命令，并且 sudo -l 的输出中包含 env_keep+=LD_PRELOAD 语句，就可能发生权限提升。这种配置允许 LD_PRELOAD 环境变量在通过 sudo 运行命令时依然保留并被识别，从而可能导致以提升的权限（root）执行任意代码。
